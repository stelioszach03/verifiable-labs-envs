# Prime Intellect Environments Hub — live

All six Verifiable Labs environments are live on the Prime Intellect
Environments Hub under the `stelioszach` account. Five are at **v0.2.0**
and one (`sparse-fourier-recovery-tools`) is at **v0.3.0** (see below).
A `verifiable-labs` organisation handle is queued for rename after org
creation.

> `sparse-fourier-recovery-tools@0.3.0` is the primitive-composition
> redesign of the tool-use env. v0.1/v0.2 exposed an `ista_tool` that
> returned the OMP oracle's answer; v0.3 replaces it with five primitive
> operators (fft, ifft, soft-threshold, compute_residual, sparsity_norm)
> so the LLM has to *compose* ISTA rather than delegate. Full rationale
> in [`results/sparse_fourier_reconciliation.md`](../results/sparse_fourier_reconciliation.md)
> and [`docs/SPRINT_1_COMPLETE.md`](SPRINT_1_COMPLETE.md) Task F.

> **v0.2.0 is the first version that actually installs from a clean venv.**
> v0.1.0 shipped with two artifacts (a `verifiers>=0.1.13` pin that rejected
> the only available dev builds, and missing `env_id` attribute on env
> instances). Both were caught and corrected during Sprint 1 Polish Task A —
> full reproducer in [`docs/PRIME_INTELLECT_VERIFICATION.md`](PRIME_INTELLECT_VERIFICATION.md).

Each Hub package is a **thin wrapper** (~30 lines) around the monorepo
at https://github.com/stelioszach03/verifiable-labs-envs. The wrapper
declares the monorepo as a Git dependency via
`[tool.hatch.metadata] allow-direct-references = true`, so installing
the Hub env transitively pulls in the full source of truth for
rewards, forward operators, LLM adapters, and conformal calibration.

## Package layout

Two parallel layouts live in the repo:

- `packages/` — 7 GitHub-subdirectory-installable packages (core + 6
  envs), meant for `pip install git+https://...#subdirectory=...`
  consumption and future PyPI publication. See
  [`packages/verifiable-labs-envs-core/README.md`](../packages/verifiable-labs-envs-core/README.md).
- `environments/` — 6 Prime-Hub-compatible env directories with the
  single-file `<env_id>.py` layout + `[tool.verifiers.eval]` config
  expected by `prime env push`. Regenerated by
  `scripts/populate_prime_envs.py`.

The `environments/` layout was required because `prime env push`
builds a wheel from `pyproject.toml` that expects `<env_id>.py` at the
same directory level (not under `src/`). The GitHub-subdirectory
packages under `packages/` retain the `src/` layout for standard
Python packaging.

## One-time setup used for first publish

```bash
pip install prime
prime login                    # OAuth in browser
# CLI prompts for a username on first push and pins it:
#   stelioszach
# From the repo root, after `scripts/populate_prime_envs.py` has written
# the six environments/ wrappers:
for dir in sparse_fourier_recovery sparse_fourier_recovery_multiturn \
           sparse_fourier_recovery_tools super_resolution_div2k_x4 \
           lodopab_ct_simplified lodopab_ct_simplified_multiturn; do
    (cd environments/$dir && prime env push)
done
```

## Known gotchas

- Prime's `env push` on first publish requires a username. The CLI prompts
  interactively; `echo <username> | prime env push` works non-interactively.
- The `verifiable-labs-envs` monorepo dep is declared as a direct Git URL.
  Hatchling rejects direct references by default; the wrappers flip
  `allow-direct-references = true` to make the wheel build succeed.
- Prime builds the wheel in an isolated venv, so the monorepo dep only
  needs to resolve at install time on the consumer side, not at push time.
